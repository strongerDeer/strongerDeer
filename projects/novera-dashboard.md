# 프로젝트 소개

이벤트 성과와 NOVERA Shop의 프론트엔드 품질을 한곳에서 확인하는 사내 대시보드. GA4·Search Console·어드민·Core Web Vitals 데이터를 연결해 비교 기준과 개선 우선순위 제공.

# 배경과 문제

신규 Shop의 이벤트 성과를 판단할 KPI와 회차별 비교 기준이 없는 상황. 오픈 시점이 다른 이벤트와 오프라인 유입을 같은 기준으로 비교하기 어려웠고, 성능·접근성·SEO·이미지 전송량은 개별 도구에 분산.

기술적으로는 GA4·Search Console·자사 어드민·외부 판매 채널(카페24)의 데이터가 식별자·집계 단위·시간대 기준이 서로 달라, 한 화면에서 비교하려면 공통 모델로의 정규화가 필요한 상태. 운영 담당자에게는 목표 도달 여부를 판단할 지표, 프론트엔드에는 배포 후 품질 회귀를 지속 감지할 기준이 필요.

# 역할

**팀 구성**: 디자인 1 · 프론트엔드 1 · 백엔드 1 · 운영 1

**담당 범위**: 측정 필요성 제안, KPI 정의, 데이터 연결 협의, 정보 구조와 차트 설계, 프론트엔드 구현·배포. 운영 담당자와 성과 기준을 합의하고, 백엔드와 GA4·어드민 데이터 연동 범위 조율. Shop 품질 지표와 주간 점검 기준 수립.

# 왜 이 기술을 사용했나

- **Next.js 16 · React 19**: 지표별 서버 조회와 필터·비교 차트 상호작용을 한 화면에서 구성
- **TypeScript**: 출처마다 다른 지표 응답을 공통 모델로 정규화하고 집계 규칙을 타입으로 제한
- **GA4 Data API · Search Console**: 행동 이벤트와 검색 유입 데이터를 공식 API 기준으로 수집
- **Recharts**: 선·막대·누적 등 표준 차트 구성이라 직접 SVG로 그리는 비용 대신 선택 — 회차별 누적 도달, 기간별 추세, 품질 지표를 동일한 시각 체계로 구현
- **AWS Amplify**: Shop과 같은 기존 인프라 조건에서 사내 접근 범위를 유지한 배포·운영

# 운영·경영진용 성과 대시보드

**사용자·목적**: 운영 담당자·경영진이 이벤트 목표 도달 여부와 유입·참여·가입 흐름, 채널별 성과(페이지뷰 PV·순방문·하루 평균 방문자 DAU·회원가입·목표 도달률)를 한 화면에서 빠르게 확인.

**화면 구성**: 기간·회차·채널 필터와 핵심 KPI 카드, 오픈 후 7/14/30일 상대기간 비교, 온·오프라인·외부(카페24) 채널 유입·전환, GA4·Search Console·어드민 데이터의 식별률·커버리지 표시.

## 기술 문제와 해결

### 문제 1. 회차별 오픈일이 달라 단순 날짜 비교가 불공정한 문제

**문제**: 단순 날짜 비교로는 먼저 열린 이벤트가 항상 유리해, 회차 간 성과를 같은 기준으로 판단할 수 없음.

**해결**: 일별 데이터를 `오픈 후 N일째`로 재인덱싱하고 동일 상대기간(첫 7/14/30일)의 누적 도달을 비교. 비교 로직을 순수 함수로 분리하고 오픈일=1일째, 누락 일자=null, 누적 정렬 규칙을 테스트로 고정.

아래 예시는 회차별 오픈 날짜를 상대 일차로 정규화하는 실제 비교 로직의 핵심을 단순화한 코드.

```ts
const daysSinceOpen = (date: string, openDate: string) =>
  Math.floor(
    (new Date(`${date}T00:00:00Z`).getTime() -
      new Date(`${openDate}T00:00:00Z`).getTime()) /
      86400000
  ) + 1;
```

**결과**: 오픈 시점이 다른 4개 회차의 도달·참여를 같은 기준으로 비교 — 1회차 이후 도달 급감을 데이터로 조기 확인.

### 문제 2. GA4 이벤트가 회차별로 분리되지 않는 문제

**문제**: `space_*` 이벤트명만으로 조회하면 여러 회차의 데이터가 섞여 회차별 참여를 집계할 수 없음.

**해결**: 이벤트명과 `pagePath`를 AND 조건으로 결합해 회차별 이벤트만 집계. 좋아요 토글은 단순 집계가 아닌 `liked - unliked` 기준으로 순좋아요 계산. 어드민 API 토큰 자동 갱신과 한국 시간(KST) 날짜 기준 통일로 반복 조회와 집계 오류 방지.

아래 예시는 이벤트 조건과 페이지 경로 조건을 함께 적용하는 GA4 필터 구성의 핵심을 비식별화한 코드.

```ts
const andPage = (eventFilter: unknown, pagePaths: string[]) => ({
  andGroup: { expressions: [eventFilter, buildPageFilter(pagePaths)] },
});
```

**결과**: 회차 성격별(댓글·투표·응모) 참여 지표를 섞임 없이 집계.

## 결과

- 이벤트 회차별 성과 비교 기준과 데이터 측정 체계 마련
- 회차 성격별(댓글·투표·응모)로 핵심 참여 지표를 각각 정의하고, 전역 가입 수치의 회차 귀속 불가·인구통계 식별률(11%) 등 데이터의 한계를 화면에 함께 노출해 지표 오독 방지
- 운영·경영진이 목표 도달과 채널별 성과를 한 화면에서 확인하는 내부 도구로 활용

## 기술 문제와 해결

### 문제 3. 실사용자 지표와 테스트 점수가 다른 문제

**문제**: Lighthouse의 실험실 측정(lab) LCP가 네트워크 스로틀과 이미지 전송량에 따라 크게 흔들려, 점수 변동이 실제 사용자 경험 악화인지 판단할 수 없음.

**해결**: 실험실 지표는 참고값으로 두고, 실사용자 측정(field) LCP·INP·CLS를 합격 기준으로 분리.

- 실사용자 LCP(주요 콘텐츠 표시 시간): 2.5s/4.0s
- INP(입력 반응 속도): 200ms/500ms
- CLS(화면 밀림 정도): 0.1/0.25
- 실험실 TBT(총 차단 시간)·번들·이미지 전송량: 회귀 감지용 보조 신호

**결과**: 실사용자 Core Web Vitals 통과 판정 — 모바일 p75 LCP 약 1.7초·INP 96ms·CLS 0으로, 측정 환경 변동에 흔들리지 않는 합격 기준 확보.

### 문제 4. 측정 노이즈를 코드 회귀로 오인하는 문제

**문제**: 같은 코드를 재측정해도 수치가 달라져, 측정 편차가 회귀 경보로 이어짐.

**해결**:

- 동일 배포본 재측정으로 실제 코드 변화와 측정 편차 분리
- 비교 기준점(baseline)을 운영 release 커밋으로 고정
- TBT 노이즈 밴드 적용으로 허위 경보 제거
- 외부 API 캐시와 주간 자동 측정으로 반복 조회 비용 축소

**결과**: 측정 편차로 인한 허위 회귀 경보 제거, 주간 점검에서 실제 코드 변화만 회귀로 판정.

## 결과

- 실사용자 Core Web Vitals 통과: **모바일 p75 LCP 약 1.7초 · INP 96ms · CLS 0**(CrUX origin, 2026-06-16~07-13 28일 창 기준 — 재직 중 최종 측정 기록)
- Lighthouse 기준 **접근성 96 · SEO 100** 확인
- 성능·SEO·이미지·번들 회귀를 주기적으로 확인하는 품질 기준 마련
