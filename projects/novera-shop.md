# 프로젝트 소개

아티스트 상품 판매와 팬덤 이벤트를 위한 커머스 서비스. 상품 탐색·상세, 주문/클레임(취소·반품·환불 요청), 운영 어드민, 캠페인 랜딩 구축 및 운영.

![](/strongerDeer/project/novera-shop_architecture.png)

사용자 화면과 운영 어드민이 하나의 백엔드 API를 공유하고, 성과·품질 지표는 별도 대시보드로 연결되는 구조. Figma 토큰과 공통 컴포넌트를 두 화면에서 함께 사용.

# 배경과 문제

전담 기획자 없이 시작한 신규 커머스. 상품 탐색부터 주문·취소·반품·운영 어드민까지 화면 정책과 상태 표현 기준이 없어, 팀이 함께 만들어가야 하는 상황.

기술적으로는 주문 상품 하나가 배송 5가지·취소 4가지·반품 6가지, 15가지 상태를 오가고, 거절 이력과 환불 확정 여부에 따라 같은 상태라도 보여줄 정보와 가능한 액션이 달라진다. 이 규칙이 정의돼 있지 않아 구매자에게는 진행 상태와 사전 안내가 부족했고, 운영자에게는 반복 확인과 수동 처리 비용이 발생.

# 역할

**팀 구성**: 디자인 1 · 프론트엔드 1 · 백엔드 1 · 운영 1

**담당 범위**: 화면 정책 정리, UI 설계, 사용자·운영 화면 개발, API 연동, 배포 후 검증까지 프론트엔드 전반. 결제·주문·취소·반품 연동 기준을 백엔드와 협의하고, 팀에서 합의한 약관·개인정보처리방침·운영방침·결제 정책을 화면 흐름과 문서로 구체화. 커밋·코드 컨벤션과 개발 요청·화면 정책 문서 정비.

## 협업 방식

기존 Jira는 개발 이슈 중심이어서 기획·디자인·프론트엔드·백엔드·운영이 함께 협업하기 어려움. 협업 도구를 Notion으로 전환하고 팀별 요청과 진행 상태를 하나의 보드에서 관리. 요청 문서는 배경·대상 화면·필요한 동작·완료 기준을 같은 형식으로 작성해 직군 간 요청과 의사결정을 한곳에서 확인 가능하게 정리.

디자인 8년 경험을 살려 [프론트엔드 ↔ 디자이너 협업 가이드](https://www.notion.so/strongerdeer/33c01a7a57f1807cb274ff83297f54c0)를 직접 작성. 단위·색상·접근성(WCAG·색약 대응)·상태별 디자인·에셋 네이밍·핸드오프 체크리스트 등 두 직군의 기준을 하나의 언어로 정리해, 요청 시 반복되던 확인·재작업을 축소.

# 왜 이 기술을 사용했나

- **Next.js 16 · React 19**: 상품 페이지의 검색 노출과 초기 렌더링, 마이페이지의 상호작용을 함께 처리하기 위한 App Router 기반 구조
- **TypeScript**: 주문·결제·클레임처럼 상태 조합이 많은 도메인의 API 응답과 화면 상태를 타입으로 제한
- **TanStack Query v5**: App Router의 fetch 캐시만으로는 클레임 처리 후 주문 도메인 단위 무효화와 무한 스크롤 페이지 상태를 다루기 어려워 선택. 서버 상태 캐싱과 mutation 이후 캐시 동기화 관리
- **Tailwind CSS**: 빠른 화면 확장과 반응형 UI 구현, 공통 상태 스타일 정리
- **AWS Amplify**: 기존 AWS 인프라에 연결된 배포 환경(주어진 조건) — 브랜치별 배포·운영 구성

- **사전 안내·진입 흐름**: 상품별 안내문구 영역으로 해외결제·예약 판매·배송 안내를 사전 노출, 로그인 진입 경로를 `/sign-in`으로 통일해 가입 후 원래 페이지 복귀

# 기술 문제와 해결

## 문제 1. 취소·반품·거절이 얽히는 주문 상태 15가지

**문제**: 주문 상품 하나가 결제완료부터 구매확정까지 배송 5가지, 취소 4가지, 반품 6가지 상태를 오간다. "거절되면 어떤 상태로 돌아가는가", "환불금액은 언제 확정되는가" 같은 규칙은 정의돼 있지 않았고, 같은 상태라도 거절 이력이 있으면 보여줄 정보와 가능한 액션이 달라진다. 화면마다 판정 기준이 다르면 그대로 구매자 문의와 운영 수동 확인 비용으로 돌아오는 상황.

**해결**:

- **정책부터 정리** — 팀에서 결정한 약관·개인정보처리방침·운영방침·결제 정책을 문서로 정리하고, 백엔드와 협의해 화면 규칙으로 확정. 취소 가능 시점은 결제완료 상태로 제한, 거절은 별도 상태가 아니라 이력으로 기록(상품은 원래 상태로 복귀), 환불금액은 취소·반품 완료 전까지 '예상 환불금액'으로 표기.
- **판정 로직 단일 출처화** — 상태 그룹(정상·취소·반품·거절) 분류, 상태 배지 색, 취소·반품·구매확정·배송조회·철회 가능 여부 판정을 한 모듈에 모아 주문 목록·주문 상세·클레임 화면이 같은 기준을 공유.
- **시나리오로 검증** — 상태별 카드·모달 흐름을 mock 데이터로 확인하는 개발 전용 시나리오 미리보기 페이지를 만들어, 실제 주문을 만들지 않고도 상태 조합을 검증.

![주문 상태 시나리오 미리보기(dev 전용) — 취소거절 탭: 거절 후 주문완료 복귀, 요청→거절 사유 이력](/strongerDeer/project/novera-shop_orders-preview.jpg)

상태별 탭으로 mock 시나리오를 확인하는 개발 전용 화면. 취소가 거절되면 상품이 주문완료로 복귀하고 요청·거절 사유가 이력으로 남는 정책이 화면에 그대로 반영된다.

아래 예시는 실제 판정 모듈의 핵심을 단순화한 코드.

```ts
// 액션 가능 여부는 화면이 아니라 단일 모듈에서 판정한다
const isCancellable = (product: OrderProduct) =>
  product.statusCode === 'ORDST_WAIT' && // 취소 가능 = 결제완료만
  getRejectInfo(product)?.type !== 'cancel'; // 취소거절 이력 상품은 재취소 불가
```

**결과**: 교환 폐지, 취소거절 상태 신설, 반품 수거중 단계 추가 같은 정책 변경을 판정 모듈 수정만으로 목록·상세·클레임 화면에 일괄 반영. 클레임 상세에는 반품배송비·환불금액 내역을 통합해 예상 환불금액을 요청 전에 확인하도록 개선.

## 문제 2. 운영자가 올린 이미지가 화면을 느리게 만드는 문제

**문제**: 배너·상품·기획전 이미지가 원본 크기로 업로드되고, 세로로 긴 상품 상세 이미지는 일괄 리사이즈 후 가로 해상도가 크게 감소.

**해결**: 어드민 업로드 시점에서 용도별 압축 정책을 분리. 썸네일·배너·본문·서류마다 최대 가로폭과 용량을 다르게 적용하고, 압축본이 더 큰 경우는 원본을 사용. Shop에서는 `next/image`와 WebP 포맷을 사용하고 첫 상품 이미지의 우선순위를 조정.

**결과**: 품질 대시보드의 동일 홈 화면 Lighthouse 전송량 기록에서 **6,645KB → 381KB(약 94.3%)** 감소 확인.

![Quality 측정 이력 — 홈 이미지 전송량이 6,645KB에서 381KB로 감소한 추세](/strongerDeer/project/novera-shop_image-trend.svg)

6월 19일 최적화 반영 후 전송량이 6MB대에서 400KB대로 낮아졌고, 이후 주간 측정에서도 유지. `shop-chart` Quality가 Lighthouse `resource-summary` 전송 바이트를 동일 홈 화면 기준으로 누적한 결과.

## 문제 3. 무한 스크롤에서 필터 후 빈 화면이 생기는 문제

**문제**: 필터 적용 후 실제로는 데이터가 있는데 빈 화면이 먼저 노출됨.

**원인 가설**: 현재 페이지의 필터 결과만으로 빈 상태를 판정해, 결과가 다음 페이지에 남아 있는 경우를 놓친다 — 페이지 단위 조회와 화면 단위 판정의 불일치.

**해결**: 다음 페이지 존재 여부와 fetch 상태를 확인해 필요한 경우에만 추가 조회하도록 정리.

- `loadedPageCount`를 ref로 기억해 같은 페이지에서 중복 fetch 방지
- `isFetching`과 `isFetchingNextPage`를 함께 확인해 연속 호출 방지
- 에러 상태와 빈 상태를 분리해 `ListErrorComponent`와 `ListEmptyComponent`를 따로 노출

**결과**: 데이터가 남아 있는 상태에서 빈 화면이 먼저 뜨는 문제와 중복 조회 제거.

## 그 외 해결한 문제

대표 사례 외에도 상태·데이터 흐름에서 아래 문제를 해결.

- **웹폰트 초기 로드 2.15MB** → 한글 `unicode-range` 다이나믹 서브셋으로 전환해 페이지당 약 320KB로 감소
- **SSR·클라이언트 캐시 경계** → TanStack Query prefetch/dehydrate로 서버 선조회 데이터를 클라이언트 캐시에 이어받아 초기 재요청 제거(`staleTime` 5분, `refetchOnMount/Focus/Reconnect` off)
- **SEO robots 병합 버그** → 하위 `robots: undefined`가 상위 `noindex`를 덮던 문제를, 값이 명시된 경우에만 병합하도록 수정해 민감 페이지 색인 차단 유지
- **어드민 다중 처리 캐시 정합성** → 상품별 상태가 다른 주문을 `useQueries` 병렬 조회하고, mutation 후 주문 도메인 query를 일괄 무효화해 목록·상세·대시보드 동기화

# 결과
