# 포트폴리오 리뉴얼 설계

> 작성일: 2026-07-09
> 근거 문서: Notion "프론트엔드 개발자 강혜진"(이력서), "포트폴리오"(공개 원칙·프로젝트 5종 상세)

## 배경 / 문제

현재 사이트(`src/data/*.ts`)는 노션 이력서보다 명백히 구버전이다.

- Hero 카피가 "0.3초의 짧은 인터랙션" — 노션 최신 포지셔닝("0.1초의 피드백을 데이터로 증명하는")과 불일치
- `careers.ts`에 NOVERA 경력이 아예 없음 — 현재 재직 중인 회사가 누락된 상태
- `project.ts`(9개 프로젝트) 중 NOVERA 관련 프로젝트가 0개 — Shop 개선, 이벤트 분석 대시보드, 품질·성능 대시보드, NDS 디자인시스템이 전부 빠짐
- Page0127이 Firebase 시절 구버전(기획/디자인/개발 100% 개인작업) — 노션은 Supabase, 소셜 기능, AI 취향 분석까지 확장된 버전이며 "개발 중인 개인 서비스"로 정직하게 포지셔닝
- Weniv Analytics 등 대시보드 프로젝트가 NOVERA 대시보드들과 메시지 중복
- 오래된 단순 퍼블리싱 프로젝트(TeemStone, Jeju Algorithm 등)가 카드 목록에서 최근 포지션과 무관하게 큰 비중 차지
- Skills가 "0.3초" 시절 기술 나열형 — NOVERA 스택(TanStack Query, Vanilla Extract, Style Dictionary, next-intl 등) 미반영

## 목표

노션 이력서/포트폴리오 문서를 단일 진실 공급원(source of truth)으로 삼아 사이트 데이터를 갱신하고, "디자인 출신 프론트엔드 개발자 · 0.1초의 피드백 · 데이터로 검증하는 UI 개선"이 자연스럽게 읽히도록 구조를 재편한다.

## 범위

- `src/data/*.ts` (introduce, careers, project, experience, skill) 전면 갱신
- `src/components/Hero.tsx`, `Skills.tsx` (TabContents 텍스트) 갱신
- `src/components/project/ProjectThumb.tsx`에 이미지 없을 때의 텍스트+수치 폴백 레이아웃 추가
- NOVERA 신규 프로젝트 4개의 상세 `projects/*.md` 신규 작성 (5단계: 문제/역할/해결/결과/공개 가능한 근거 방식)
- `page0127.md`는 "기능 나열"이 아닌 "UI/UX 판단 과정" 구조로 전면 재작성 (기존 8단계 배경/기획/기술스택/기능/트러블슈팅/성과/인사이트/향후계획 포맷 폐기 — 아래 2-2 참고)
- 오래된 프로젝트(WeniVooks, Weniv Bootcamp, Weniv Link, Jeju AI Conf, Jeju Algorithm, TeemStone) md는 카드 노출 문구만 다듬고 md 본문은 유지(상세 진입 시 그대로 보임 — 삭제하지 않음)
- Weniv Analytics는 Other Projects로 이동(대시보드 메시지 중복이 이유이며 삭제하지 않음)

범위 밖: 디자인 시스템(CSS/Tailwind 토큰) 자체 변경, 애니메이션 라이브러리 교체, 배포 파이프라인.

## 섹션별 설계

### 1. Hero (`Hero.tsx`, `introduce.ts`)

- 메인 카피: "0.1초의 피드백을 데이터로 증명하는 강혜진 개발자입니다." (노션 원문)
- 짧은 소개 2~3문장: 0.1초의 정의 + NOVERA Shop에서 하는 일(상품 탐색/주문·클레임/SEO/속도지표/품질 대시보드) — 노션 원문 문장을 그대로 재사용
- `STRENGTHS`(웹디자인 출신 / 지식공유 / UX개선 3블록) → 핵심 키워드 3개로 교체: **UI 피드백 · 성과/품질 대시보드 · 디자인 시스템**. 각 키워드에 노션의 "경력" 콜아웃 3줄(커머스 운영 화면 개발자 / 성과를 정리하는 개발자 / 디자인과 개발을 함께 이해하는 개발자)을 짧은 설명으로 매칭
- 코드 타이핑 애니메이션(`De{sign||velop}er`)은 유지 — 요구사항과 무관한 브랜드 요소

### 2. Featured Projects (`project.ts`, `ProjectCard.tsx`, `ProjectThumb.tsx`)

Featured 5장, 모두 독립 카드:

1. **Page0127** — 대표 공개 프로젝트. 기존 이미지 자산(책장/차트/goal 등) 그대로 활용. 카드 설명 2~3줄은 "기록→통계→취향분석→추천" 흐름 강조, "개발 중인 개인 서비스"로 명시
2. **NOVERA Shop 사용자/운영자 화면 개선** — 실무 성과. 이미지 없음 → 텍스트+수치 카드(79.4% SNS 간편가입, 94% 이미지 경량화, LCP 1.44s). 상세페이지는 "설계 판단 기준" 톤(아래 2-1 참고)
3. **이벤트 성과 분석 대시보드** — 실무 성과. 텍스트+수치 카드(DAU 목표 대비 122%, 대시보드 6종)
4. **품질·성능 모니터링 대시보드** — 실무 성과. 텍스트+수치 카드(LCP 1.44s·INP 88ms·CLS 0, 접근성 96·SEO 100)
5. **NDS 디자인 시스템** — 디자인-개발 연결. 텍스트+수치 카드(컴포넌트 30종, npm 패키지 배포)

`ProjectThumb.tsx`에 `thumb`이 빈 문자열일 때 이미지 대신 표시할 폴백 UI 추가: 프로젝트 아이콘 + 핵심 수치 1~2개를 크게 보여주는 레이아웃(신규 마크업, 기존 `.thumb` 클래스 컨텍스트 안에서 조건 분기).

카드 설명(`description`)은 각 프로젝트 노션 "문제" 문장을 1문장으로 압축, `metrics`(카드에 노출되는 bullet 2~3개)는 노션 "결과" 섹션의 수치를 그대로 사용.

#### 2-1. NOVERA Shop 상세 — "데이터 없는 초기 단계"를 판단 근거로 리프레이밍

서비스 오픈 후 2~3개월이라 A/B 테스트나 사용자 행동 데이터로 UX 개선을 증명하기 어렵다는 현실이 있다. 이를 감추지 않고, "측정 데이터가 아직 없는 시점에 무엇을 근거로 설계했는가"를 보여주는 섹션으로 정면 배치한다.

- 성능/SEO 수치(94% 경량화, LCP 1.44s 등)는 결과 섹션에 그대로 유지 — 이건 실측 가능한 사실이므로 삭제하지 않음
- 별도로 **"설계 판단 기준"** 소섹션을 추가해 상태·로딩·오류·빈 화면 설계에 적용한 기준을 서술형으로 정리. 예: 주문 상태별 안내문구를 사전 노출한 이유(CS 발생 가능성을 구조적으로 줄이기 위함 — 이미 노션에 있는 근거), 다중 상품 일괄처리 UI를 만든 이유(운영자 반복 작업 비용 감소) 등 **노션 원문에 이미 있는 "왜 이렇게 만들었는가" 근거를 재료로 삼는다** — 없는 사용자 데이터나 A/B 결과를 지어내지 않는다
- 톤: "데이터가 쌓이면 검증하겠지만, 지금은 이런 기준으로 판단했다"는 정직한 1인칭 서술. 방어적("아직 부족합니다")이 아니라 능동적("이 시점엔 이런 원칙을 우선했다")으로 쓴다

#### 2-2. Page0127 상세 — "기능 나열" 대신 "UI/UX 판단 과정" 구조

기존 8단계(배경/기획/기술스택/기능/트러블슈팅/성과/인사이트/향후계획) 포맷을 폐기하고, 화면 단위로 아래 3단 구조를 반복하는 형태로 재작성한다.

1. **사용자 목표** — 이 화면에서 사용자가 하려는 것 (예: "내가 읽은 책을 한눈에 훑어보고 싶다")
2. **UI 판단** — 그래서 어떤 레이아웃/상태/피드백을 선택했는가 (예: 완독은 책등 방향, 인생 도서는 표지 노출 — 물리적 책장의 인지 습관을 그대로 차용)
3. **판단 근거** — 그 판단의 근거(사용성 원칙, 데이터 구조상 제약, 기존 서비스 벤치마크 등)

우선 노출 화면은 **내 책장 · 독서 통계 · AI 취향 분석**(캡처 우선순위 1~3위와 동일) 3장을 기본으로 하고, 랜딩/홈(4위)은 확보 여유가 있을 때 추가한다. 각 화면마다 위 3단 구조를 짧게(화면당 4~6줄) 적용한다.

"개발 중"이라는 방어적 표현은 본문 곳곳에 흩어놓지 않고, 문서 맨 끝 **로드맵** 섹션 한 곳으로 모은다 — 구현 완료 / 구현·개선 중 / 확장 기능(노션 "구현 범위" 표 3분류)을 여기서만 다룬다. 본문 화면 설명에서는 "개발 중이라"는 식의 전제를 반복하지 않고 완료된 판단으로 서술한다.

md의 heading 구조가 바뀌므로 `src/utils/projects.ts`의 `adjustHeadingLevel()`에 있는 section1~8 id 매핑(🤔 배경, 📝 기획 등 고정 문자열 매칭)도 Page0127의 새 heading에 맞춰 갱신하거나, 해당 id 부여 로직이 다른 프로젝트 md에는 영향 없음을 확인 후 Page0127용 heading 텍스트만 매핑 규칙에 맞게 조정한다.

## 에러 처리 / 엣지 케이스

- `ProjectThumb`에서 `thumb`이 빈 문자열인 카드는 `Image` 렌더링을 건너뛰고 폴백 UI만 표시 — 기존 `{thumb && <Image .../>}` 조건이 이미 이를 지원하므로 폴백 마크업만 추가하면 됨
- 신규 md 4개의 frontmatter는 기존 `I_PROJECTS` 인터페이스를 그대로 따름(새 필드 추가 없음) — `url`이 없는 내부 프로젝트는 `url: ""`로 두고 `ProjectBtns`가 이미 `{url && ...}` 조건부이므로 버튼 미노출 처리됨. `github`도 동일하게 optional 유지

## 최종 검수 체크리스트 (콘텐츠 작성 완료 후 적용)

전체 텍스트 작성이 끝나면 채용 담당자 시점에서 아래 4가지를 점검한다:

1. **메시지 일관성** — "디자인 감각 + 데이터 검증" 하이브리드 포지셔닝이 Hero부터 각 프로젝트 상세까지 끊기지 않고 이어지는가
2. **구조 준수** — NOVERA 4종 프로젝트가 문제→역할→해결→결과(→공개 가능한 근거) 구조를 실제로 지키는가, Page0127이 사용자목표→UI판단→판단근거 구조를 지키는가
3. **협업 서술** — "디자이너와 협의", "백엔드와 협의" 같은 문장이 매 항목에 기계적으로 반복되지 않고 실제 맥락에 맞게 자연스러운가
4. **AI 상투어 제거** — "혁신적인", "완벽한", "다양한 노력을 기울였습니다" 류의 공허한 수식어, 근거 없는 강조 표현을 걸러낸다

이 체크리스트는 설계 문서 승인 후 콘텐츠(md, 데이터 파일) 작성이 끝난 시점에 한 번 전체 재독 형태로 실행한다.
